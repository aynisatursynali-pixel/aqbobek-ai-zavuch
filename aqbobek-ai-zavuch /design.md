# Design Plan: Aqbobek AI Zavuch

## Goal

Сделать интерфейс не "табличкой для айтишников", а **операционным штабом директора школы**, который:

- быстро показывает, где горит;
- не требует изучения сложных экранов;
- помогает принимать решения по attendance, заменам, инцидентам и документам;
- выглядит как продукт для управления школой, а не как generic admin panel.

## Visual Direction

### Core mood

- Тёплый, человеческий, управленческий интерфейс.
- Ассоциация не с "CRM", а с **школьным штабом + аккуратным бумажным журналом + современным AI-слоем**.
- Светлая палитра для читаемости в рабочее время.

### Palette

- `Background`: тёплый ivory / paper.
- `Primary`: глубокий teal.
- `Secondary`: amber / sand.
- `Alert`: muted red.
- `Neutral text`: глубокий зелёно-графитовый.

### Typography

- Заголовки: `Fraunces` для ощущения значимости и "официальности".
- Интерфейсный текст: `Manrope` для плотной, современной читаемости.

## Screen Architecture

### 1. Command Center

Главный экран директора.

Содержит:

- KPI по школе на сегодня;
- dining summary;
- attention queue;
- heatmap нагрузки педагогов;
- preview расписаний администрации и сервиса;
- статус AI-модели.

### 2. Attendance Center

Экран под утреннюю рутину.

Содержит:

- общую сводку по посещаемости;
- missing reports;
- карточки по каждому классу;
- готовность заявки в столовую.

### 3. Operations Desk

Экран быстрых действий.

Содержит:

- AI intake textarea;
- voice trigger;
- quick prompt chips;
- результат structured extraction;
- задачи;
- инциденты;
- замены.

### 4. Schedule Hub

Экран для завуча и директора.

Содержит:

- фильтры по `day / class / teacher / room`;
- тепловую карту нагрузки;
- сетку уроков;
- персональные планы staff.

### 5. Document Lab

Экран для бюрократического copilot.

Содержит:

- выбор приказа;
- текст запроса;
- retrieved regulation pack;
- plain-language summary;
- missing information;
- checklist;
- готовый черновик.

## UX Principles

### 1. One glance first

Каждый экран должен отвечать на вопрос за 3-5 секунд:

- Что сейчас важно?
- Где блокер?
- Какой следующий шаг?

### 2. Explain, then act

AI не просто выполняет действие, а показывает:

- что понял;
- какой intent определил;
- что собирается создать;
- что требует подтверждения.

### 3. Director-friendly wording

Все тексты в UI должны быть ближе к управленческому языку школы:

- "Не отчитались";
- "Нужна реакция";
- "Замена рассчитана";
- "Черновик приказа";

вместо сухих "records", "entries", "entities".

### 4. Calm over clutter

Даже при большом объёме данных экран не должен визуально шуметь.

Решения:

- крупные карточки;
- воздух между блоками;
- ограничение на число акцентов;
- мягкая цветовая иерархия вместо кислотных статусов.

## Component Plan

### Navigation

- Sticky left rail on desktop.
- Прямые названия модулей без техно-жаргона.

### Metric cards

- Большое число.
- Короткий label.
- Одна строка объяснения "почему это важно".

### Status badges

- `good`
- `warn`
- `bad`
- `neutral`

Их цвет должен быть стабилен по всему интерфейсу.

### Queue cards

Для задач, инцидентов и замен:

- короткий title;
- 2-3 ключевых поля;
- текущий статус;
- одно действие.

### Heatmap cards

Не классическая matrix-grid, а практичный executive format:

- имя педагога;
- max lessons/day;
- longest streak;
- визуальная полоска нагрузки.

### Result shells

Для AI и документов:

- structured sections;
- не raw JSON как основной UI;
- JSON можно оставить как debug block, но вторично.

## Motion

Использовать только полезную motion-составляющую:

- плавный вход страницы;
- hover lift у карточек;
- быстрый tactile response у кнопок;
- без декоративной перегруженности.

## Accessibility

- Контраст текста выше, чем в типичных glassmorphism-макетах.
- Кликабельные элементы крупные.
- На мобильном всё должно складываться в одну колонку.
- Важные действия не прячутся в иконки без подписи.

## Content Strategy

Нужно различать три уровня текста:

### Level 1: Executive

Для директора:

- коротко;
- по делу;
- с фокусом на решение.

### Level 2: Operational

Для завуча и админ-блока:

- кто;
- когда;
- что именно сделать.

### Level 3: AI / Legal

Для document lab и AI result:

- explainable output;
- checklist;
- missing info;
- source links.

## Next Design Iterations

### Iteration 1

- внедрить текущую дизайн-систему;
- стабилизировать карточки, таблицы и mobile layout.

### Iteration 2

- добавить drag-and-drop сценарий для сетки;
- визуализировать конфликт кабинета/учителя;
- сделать timeline по дням.

### Iteration 3

- отдельный экран "AI suggestions";
- comparison mode "до / после замены";
- printable document preview.

## Success Criteria

Дизайн можно считать удачным, если:

- директор понимает состояние школы без обучения;
- демонстрация идет по понятному story flow;
- жюри видит не только AI, но и **готовый инструмент управления**;
- интерфейс поддерживает логику ТЗ, а не просто украшает данные.
