# Аудит покрытия ТЗ и план развития MVP

## 1. Краткий вывод

Текущее состояние проекта можно считать **ранним функциональным MVP**, который уже закрывает базовую демонстрацию по 3 направлениям:

- сбор посещаемости из сообщений;
- регистрация инцидентов и создание задач;
- подбор замены по расписанию на основе простых правил.

По моему разбору, **в среднем покрыто около 35-45% MVP-требований из ТЗ**, если считать именно наличие работающей функции в коде, а не только UI-заглушки.

Лучше всего покрыты:

- Telegram-бот как интерфейс для учителей;
- attendance summary для столовой;
- базовый task manager;
- простая логика инцидентов;
- демонстрационный сценарий замены учителя.

Слабее всего покрыты:

- полноценный scheduler / генерация расписания с нуля;
- "ленты" и групповые слоты;
- расписание для всех сотрудников, не только учителей;
- real AI / LLM orchestration;
- RAG по приказам;
- продакшн-архитектура, БД, роли, уведомления и аудит изменений.

## 2. Что уже реализовано по ТЗ

### 2.1 Web-dashboard директора

Есть рабочий dashboard в [frontend/index.html](/Users/macbook/Desktop/aqbobek-ai-zavuch/frontend/index.html), в котором уже присутствуют:

### 2.2 Telegram-бот

В [backend/bot.py](/Users/macbook/Desktop/aqbobek-ai-zavuch/backend/bot.py) уже реализован Telegram-бот, что напрямую соответствует обязательному требованию ТЗ по мессенджеру.

- нет отправки персональных уведомлений конкретным учителям по реальным контактам.

## 4. Главные сильные стороны текущего решения

- Уже есть **сквозной demo flow**: Telegram -> parsing -> JSON storage -> dashboard.
- Проект опирается на **реальные данные школы** из Excel, а не полностью синтетический UI.
- Замены уже основаны не на ручной кнопке, а на **логике поиска кандидата по предмету и занятости**.
- Attendance, incidents и tasks связаны между собой достаточно хорошо для хакатонного прототипа.
- Для pitch уже можно показать, что решение снимает часть WhatsApp/Telegram-рутины у директора.

## 5. Главные разрывы и что стоит улучшить в первую очередь

### 5.1 Самый критичный разрыв: это не AI-оркестратор, а rule-based MVP

Сейчас проект выглядит как умный workflow tool с regex-логикой, а не как полноценный AI assistant.

Что улучшить:

- добавить LLM-слой для разбора attendance, инцидентов и задач;
- ввести structured extraction в JSON через одну унифицированную схему;
- сохранять confidence и raw input;
- показывать в UI, что именно "понял" AI и что требует подтверждения.

### 5.2 Самый сильный шанс на баллы: усилить блок расписания

По ТЗ и критериям оценки smart scheduling даёт 30%, поэтому именно сюда выгоднее всего инвестировать время.

Что улучшить:

- сделать хотя бы **constraint-based rescheduler для замен**, а не только поиск одного свободного учителя;
- добавить учёт кабинета, лимита нагрузки в день и совместимости предмета;
- показать персональное расписание по выбранному сотруднику;
- добавить страницу "School Grid" с фильтрами `day / class / teacher / room`.

Даже если не строить расписание "с нуля", можно честно позиционировать:

- MVP уже умеет ingest + inspect schedule;
- при отсутствии учителя делает локальную оптимизацию и предлагает лучшую замену;
- next step: full schedule generator.

### 5.3 RAG нужно хотя бы начать, иначе 4-й модуль пустой

Минимально жизнеспособная версия для demo:

- загрузить 2-3 PDF/текста приказов;
- сделать retrieval по chunks;
- добавить форму "подготовь приказ/отчёт";
- задавать 3-5 уточняющих вопросов;
- генерировать черновик документа.

Даже простой RAG даст сильный demo-эффект, потому что сейчас этот модуль отсутствует полностью.

### 5.4 Перевести данные из JSON в нормальную БД

JSON-файлы подходят для демо, но мешают масштабировать:

- нет конкурентной записи;
- нет связей между сущностями;
- трудно хранить историю;
- трудно реализовать роли и уведомления.

Лучший следующий шаг:

- PostgreSQL + SQLAlchemy / Prisma;
- таблицы: `employees`, `classes`, `rooms`, `schedule_slots`, `tasks`, `incidents`, `attendance_reports`, `substitution_requests`, `notifications`, `documents`.

### 5.5 Улучшить UX под директора школы

Сейчас UI уже приятный, но ещё не похож на операционную панель школы.

Что улучшить:

- отдельная вкладка полного расписания;
- фильтр "по сотруднику";
- heatmap занятости и перегруза;
- журнал последних событий;
- кнопка "подтвердить все attendance";
- блок "кто не ответил до 09:00";
- action center с быстрыми сценариями: "учитель заболел", "создать приказ", "инцидент", "собрать столовую".

## 6. Рекомендуемый план работ

### Этап 1. Упаковать текущий MVP для сильного demo

Цель: за 1 шаг повысить понятность и ценность без тяжёлой перестройки.

Задачи:

1. Добавить `README.md` с инструкцией запуска backend, frontend и bot.
2. Вынести конфигурацию в `.env` (`BOT_TOKEN`, `OPENAI_API_KEY`, `PORT`).
3. Сделать seed/reset сценарий для демо-данных.
4. Добавить страницу/секцию "Full Schedule".
5. Добавить фильтры по `teacher`, `class`, `day`, `room`.
6. Привести тексты интерфейса к единому языку и стилю.
7. Подготовить 3-4 scripted demo scenarios.

Ожидаемый эффект:

- решение проще показать жюри;
- меньше шанс сломаться на демо;
- появляется более цельная история продукта.

### Этап 2. Усилить AI-парсинг и task orchestration

Цель: чтобы решение выглядело именно как AI assistant, а не набор regex.

Задачи:

1. Подключить LLM для extraction attendance / incidents / director tasks.
2. Ввести единый JSON schema output для всех входящих сообщений.
3. Добавить confidence score и флаг `needs_review`.
4. Добавить нормализацию ФИО сотрудников по справочнику.
5. Поддержать voice upload / transcription на backend, а не только в браузере.
6. Автоматически отправлять персональную задачу исполнителю в Telegram.

Ожидаемый эффект:

- лучшее покрытие критерия "извлечение данных";
- заметно возрастает ощущение AI-системы.

### Этап 3. Сделать сильный Smart Substitution MVP

Цель: закрыть самый "дорогой" по баллам кусок.

Задачи:

1. Ввести модель ограничений: квалификация, кабинет, дневная нагрузка, занятость.
2. Добавить ranking кандидатов по нескольким факторам, а не только weekly load.
3. Сохранять причину выбора и причину отказа по каждому кандидату.
4. Добавить fallback-сценарии:
   - свободный предметник;
   - смежный педагог;
   - администратор/завуч;
   - supervised self-study.
5. Добавить экран персонального расписания сотрудника.
6. Добавить визуальную подсветку конфликтов.

Ожидаемый эффект:

- сильнее демонстрируется "умное" управление расписанием;
- появляется ценность даже без full solver.

### Этап 4. Минимальный RAG по приказам

Цель: закрыть пустой модуль ТЗ и усилить вау-эффект.

Задачи:

1. Собрать тексты приказов №76, №110, №130.
2. Разбить на чанки и индексировать.
3. Сделать backend endpoint для вопроса по документам.
4. Добавить сценарии:
   - "подготовь приказ";
   - "подготовь отчёт";
   - "объясни простыми словами".
5. Реализовать уточняющие вопросы перед генерацией.
6. Показывать использованные фрагменты документа как обоснование.

Ожидаемый эффект:

- появляется полноценный 4-й модуль;
- проект выглядит гораздо ближе к заявленному кейсу.

### Этап 5. Нормализовать архитектуру

Цель: превратить хакатонный прототип в расширяемую систему.

Задачи:

1. Перевести хранение из JSON в PostgreSQL.
2. Разделить backend на слои: API, services, repositories, schemas.
3. Разбить фронтенд на компоненты.
4. Добавить базовую авторизацию и роли.
5. Сделать журнал действий и историю изменений.
6. Подготовить деплой.

## 7. Приоритеты, если времени мало

Если времени до защиты немного, я бы шёл в таком порядке:

1. **Усилить demo и UX**, чтобы уже существующее выглядело цельно.
2. **Подключить LLM extraction** для attendance / tasks / incidents.
3. **Улучшить smart substitution** с ограничениями и понятным объяснением выбора.
4. **Сделать минимальный RAG**, хотя бы на 1-2 сценария.
5. Только потом думать о полном генераторе расписания с нуля.

## 8. Честная формулировка для защиты

Если позиционировать проект перед жюри прямо сейчас, лучше говорить так:

> У нас уже есть рабочий MVP цифрового AI-завуча для операционной рутины школы: сбор attendance, регистрация инцидентов, постановка задач и автоматический подбор замен по расписанию. Следующий этап развития — full scheduling engine и RAG-модуль по приказам.

Это звучит честно и при этом сильно, потому что текущий репозиторий уже реально демонстрирует ценное ядро продукта.
